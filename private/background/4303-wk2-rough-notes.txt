Week 02: Evaluating and Choosing Your Tools
Evaluating and Choosing Your Tools
Last week you surveyed the modern tool landscape — editors, AI assistants, frameworks, hosting platforms. This week, you narrow the field. You'll pick the tools you'll actually use for your project this semester.

This isn't about finding the "best" tools. It's about finding the right tools for what you're building and how you work.

Start with Your Project, Not the Tools
The most common mistake in tool selection is picking tools first and then figuring out what to build with them. That's backwards. Your project determines your tools, not the other way around.

Ask yourself:

What am I building? A portfolio site has different tool requirements than an interactive web app. A blog needs different capabilities than a data visualization project.
How complex is it? If your project is primarily content and layout, plain HTML and CSS with a simple deployment pipeline might be all you need. If you want dynamic behavior, you'll need tools that help you add interactivity.
What do I already know? You know HTML and CSS. You've used VS Code and GitHub Desktop. Starting from what you know and adding tools as needed is a stronger strategy than switching everything at once.
Criteria for Evaluating Tools
When you look at a tool, consider these questions:

Does it solve a real problem for my project? If you can't name the specific problem a tool solves, you probably don't need it. "Everyone uses it" is not a reason. "It handles responsive image optimization, which my portfolio needs" is.

Can I get productive with it in a reasonable timeframe? Some tools have steep learning curves. If a tool requires weeks of study before you can use it effectively, it might not be the right choice for an 8-week course — even if it's powerful.

Tools exist in an ecosystem, so check compatibility. Your editor, your version control, your hosting platform, and your framework all need to work together. Verify that before committing time.

Consider what happens when you get stuck. Every tool has a community, documentation, and learning resources — some more than others. Tools with strong documentation and active communities are easier to troubleshoot.

Finally, check pricing. Many tools offer free plans that cover everything a student project needs. Don't invest time learning a paid tool if a free alternative does the same thing.

Making Decisions Under Uncertainty
You won't have complete information when you choose your tools. That's normal. You're making decisions based on what you know right now, and you might learn things later that change the picture.

A few principles for deciding when you're unsure:

Default to simpler. If you're torn between two options, pick the simpler one. You can always add complexity later. Removing complexity is harder.

Pick a tool, use it, and evaluate. If it's not working after a genuine effort, switch. But give it a real chance first — frustration during the learning phase is normal and doesn't mean the tool is wrong.

AI assistants are genuinely useful for tool evaluation. You can describe your project and ask which tools might fit. You can ask a tool to explain itself. You can paste an error message and get help. Use them.

Pay attention to what others in the class are using, too. If someone is building something similar to your project and using a tool you hadn't considered, that's useful information.

If the choices feel overwhelming, here's a safe default: VS Code as your editor, plain HTML and CSS for your project, and GitHub Pages for hosting. You've used these tools in earlier coursework, so you can build on what you already know. You can always add frameworks, AI tools, or other platforms later as your project grows.

A Practical Approach to Tool Selection
Here's one way to approach tool selection for your project:

Define what you're building. Write one sentence describing your project and what it needs to do.
List what you need. What capabilities does your project require? Content display? Interactivity? Data from external sources? A specific visual style?
Match tools to needs. For each need, identify one or two tools that could address it. Use what you learned in Week 01's tool landscape survey.
Check compatibility. Make sure your chosen tools work together. If you're using a static site generator, does it deploy to your chosen hosting platform?
Set up and test. Install your tools, create a test project, and make sure the pipeline works end-to-end: edit, save, commit, deploy.
You'll go through this process in Exercise 02 and formalize your choices in Assignment 01.

Key Takeaways
Let your project drive tool selection, not the other way around
Evaluate tools against real criteria: does it solve a problem, can you learn it, does it play well with your other tools
Default to simpler when uncertain — you can add complexity later
Tool choices aren't permanent — commit, evaluate, and adjust if needed
Week 02: Defining Your Project
Defining Your Project
This semester, you'll build a single web project that evolves across eight weeks. You'll start with a concept, build a working version by Week 05 (Project 01), and then extend and refine it into a final MVP by Week 08 (Project 02). The project you define this week is the one you'll work on for the rest of the course.

That means the concept matters. Not because it needs to be perfect — it will change as you build — but because it sets the direction for everything that follows.

What Makes a Good Project Concept
A good project for this course has four qualities:

It has a clear purpose. You can explain in one or two sentences what the project does and why it exists. "A portfolio site that showcases my design work to potential employers" is clear. "A website about stuff" is not.

It has an identifiable audience. Someone will use this thing. Who? A hiring manager reviewing your work? A community of hobbyists sharing resources? Your classmates looking for study tools? Knowing your audience shapes every decision you make — from content to layout to features.

It's buildable in eight weeks. You'll have about six weeks of active building time (Weeks 02-04, then Weeks 06-07). Your concept needs to be achievable within that timeframe, given your current skill level and the tools available to you. Ambitious is good. Unrealistic is not.

It interests you. You'll spend the rest of the semester working on this. Pick something you actually want to build. Boredom is the enemy of good work.

Types of Projects That Work Well
You have wide latitude in what you build. Here are some categories that work well for this course:

Portfolio or personal site. A site that presents your work, skills, or identity. This could be a design portfolio, a professional site, a creative showcase, or a personal brand page. Good fit if you want to focus on visual design, layout, and content presentation.

Content-focused site. A blog, resource collection, tutorial site, or fan wiki. This type leans heavily on content organization, navigation, and information architecture.

Interactive tool or app. A calculator, quiz, planner, generator, or utility. Works well if you want to explore interactivity and dynamic behavior. Remember: you don't need to write JavaScript from scratch — your tools can help.

Community or event site. A site for a club, organization, event, or cause. The focus here is audience engagement, calls to action, and real-world utility.

Experimental or creative project. An art piece, interactive story, data visualization, or something that doesn't fit a standard category. Consider this if you want to push boundaries and explore what the web can do.

Any of these can be scoped appropriately for this course. The key is matching your concept to a scope you can deliver.

Scoping Your Project
Scope is the most common challenge in project-based courses. Projects that are too ambitious lead to frustration and incomplete work. Projects that are too simple don't give you enough to learn from.

For Project 01 (Week 05), aim for a working foundation: the core structure is in place, the site is deployed and accessible via URL, and the basic functionality works. It doesn't need every feature. It needs to stand on its own as a usable, deployed thing.

For Project 02 (Week 08), you'll extend P01: add depth, improve functionality, polish the experience, and finalize documentation. P02 is P01 made better, not a different project.

A practical scoping exercise:

Write down everything you'd want your project to do if you had unlimited time.
Circle the items that are essential — the things that make it the project you described.
That's your P01 scope. Everything else is P02.
If your "essential" list still feels too long, cut more. You can always add features later. You can't get time back.

Defining Purpose and Audience
Your project proposal (Assignment 01) will ask you to articulate your project's purpose and audience. Here's how to think about each:

Purpose is what the project does and why it exists. Be specific. "Help people learn about Texas wildflowers" is better than "an educational website." "Give my band an online presence with upcoming shows and music samples" is better than "a music site."

Audience is who will use it and what they need. Think about what your audience knows, what they're looking for, and how they'll interact with your site. A portfolio site for hiring managers has different needs than a resource site for fellow students.

Purpose and audience together drive your design decisions. They determine what content to include, how to organize it, what features matter, and what can be left out.

Key Takeaways
Your project concept sets the direction for the entire semester — pick something with clear purpose, identifiable audience, buildable scope, and genuine interest
P01 is a working foundation; P02 is that foundation extended and polished
Scope aggressively — essentials for P01, everything else for P02
Purpose and audience drive design decisions — define them clearly in your project proposal

Assignment 01: Project Proposal
Overview
Your project proposal is where your semester-long project begins. This assignment asks you to define what you want to build, why it matters, what tools you plan to use, and what challenges you expect to face. A strong proposal sets the direction for everything that follows — your prototype in Project 01 and your final MVP in Project 02.

Points: 40

Due: Sunday, March 29 by 11:59 PM CT

Instructions
Write your proposal and submit it as a PDF or Markdown file (uploaded to Canvas or as a GitHub URL). Your proposal should address each of the following four sections.

1. Project Description (10 points)
Describe your project clearly and specifically. Your description should answer three questions:

What is it? Describe the site or application you want to build. Be specific about what it does and what a visitor experiences.
Why does it matter? Explain why this project is worth building. What problem does it solve, or what purpose does it serve?
Who is it for? Identify your intended audience. Who would use this, and why would they care?
Specificity matters here. "A website about food" tells your instructor nothing. "A recipe site for college students who want to cook meals under $10 with five or fewer ingredients" tells a clear story about what you are building and for whom.

2. Feature Plan (10 points)
List the features you plan to include in your project, organized into two tiers:

Project 01 (due Week 05): The features you will have working in your first prototype. These are the essentials — the minimum set of features that make your project functional and demonstrable.
Project 02 (due Week 08): The features you plan to add, improve, or refine for your final MVP. These build on the P01 foundation and represent the complete vision for your project.
For each feature, write one sentence describing what it does. Keep descriptions concrete — "a filterable gallery of project images" is better than "some kind of portfolio section."

3. Tools and Technologies (10 points)
Identify the tools and technologies you plan to use across the following categories. For each, explain why you chose it over alternatives.

Code editor or development environment: What will you use to write and edit your project files?
Framework, platform, or approach: Are you writing HTML/CSS from scratch, using a static site generator, working with a website builder, or something else?
Hosting and deployment: Where will your site live? How will you publish it?
AI tools: If you plan to use AI assistants, which ones and for what tasks?
Other tools: Anything else — design tools, image editors, version control interfaces, libraries, APIs, or services.
4. Risks and Unknowns (10 points)
Identify at least two risks or unknowns that could affect your project. These might include:

A feature you are not sure how to build yet
A tool you have not used before
A dependency on an external service or API
A scope concern — whether you can realistically finish what you are planning
Honesty matters more than confidence here. Every project has unknowns. Naming them early is a sign of good planning, not weakness.

Submission
Submit your proposal as a PDF or Markdown file.
