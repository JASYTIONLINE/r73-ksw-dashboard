Week 03: Building Your Project Structure
Building Your Project Structure
You have a concept, and you've chosen your tools. Now it's time to build something. This page is about structuring your project by organizing files, creating pages, and establishing a foundation that everything else will build on.

Think in Pages and Sections
Every web project is made of pages, and every page is made of sections. Before you write a single line of code, think about what pages your project needs and what each page should contain.

For most projects, this starts with a few basic questions:

What pages does your project need? A home page is a given. What else? An about page? A portfolio or gallery? A contact page? Individual project pages? A blog? List them out.
What does each page contain? For each page, identify the major sections. A home page might have a hero section, an introduction, featured content, and a footer. A portfolio page might have a grid of project cards.
How do users move between pages? Navigation is the connective tissue of your project. Think about your main navigation (what links appear on every page) and how users flow from one page to the next.
Write this down before you build. A simple outline saves time by giving you a plan to follow.

File Organization
How you organize your files depends on your tools, but a few principles apply across the board:

Keep it flat until you need depth. Start with a simple structure. If your project has five pages, you might not need subdirectories at all. Add folders when they serve a clear purpose, such as grouping images, separating components, and organizing content by category.

Name files descriptively. about.html is better than page2.html. Future you should be able to find any file by its name alone.

Separate content from assets. Keep your HTML, CSS, images, and other assets organized. A common pattern:

project/
├── index.html
├── about.html
├── portfolio.html
├── css/
│   └── styles.css
├── images/
│   ├── hero.jpg
│   └── profile.jpg
└── js/
    └── main.js
If you're using a framework or static site generator (which many students will), it will have its own file structure conventions. Follow them. The framework or static site generator's getting-started docs will explain the file structure conventions, which exist for good reasons.

Building the Skeleton First
Professional developers rarely build a page top-to-bottom, finishing each section before starting the next. Instead, they build the skeleton first: the structure without the polish. Scaffolding is a common approach.

HTML structure. Write the semantic HTML elements for your page, such as headers, navigation, main content areas, footers, and sections. Don't worry about how it looks yet. Get the content hierarchy right. (Yes, even if you plan to use a framework, you need to get your content hierarchy right.)
Content placement. Put your actual content in place, or use realistic placeholders. Placeholder text is fine for now, but real headings and navigation labels help you see whether your structure makes sense. Use real content whenever possible.
Basic styling. Add enough CSS to make the page usable (e.g., basic layout, typography, spacing). It doesn't need to look finished, but it does need to look intentional.
Navigation. Wire up your navigation so pages link to each other. Test every link. Even if, initially, every page is black text on a white background without any layouts. Test your page links. Broken navigation is a fast way to make a project feel incomplete.
This approach lets you see the whole project taking shape before you invest time in polishing any one piece.

Working with Your Tools
If you're using plain HTML and CSS, the structure above is yours to create. If you're using a framework or site generator, much of this structure is handled for you! However, you need to understand what it's doing. Look at the starter templates and their corresponding GitHub repositories to understand how the structure works.

Frameworks provide templates, components, and routing. You define your content, and the framework handles how it's assembled and served. If you're using one, spend time understanding its file structure and how to add new pages. Again, look at the templates, the code, and review the corresponding GitHub repositories.

AI assistants can help you scaffold a project quickly. Describe what you need ("Create a three-page portfolio site with a header, navigation, and footer on each page") and use the generated code as a starting point. Review what it produces. Have a fast review loop to make sure things are working. Make sure you understand the structure before building on it.

Templates and starters give you a pre-built foundation. Many frameworks and hosting platforms offer starter templates for common project types. Starting from a template is a valid approach. Seriously. The key, though, is to make sure you customize any starter template enough that it becomes yours and does not look or feel like a starter template with barely any customization.

Whatever tools you use, the goal this week is the same: a real project with real pages, real content (or realistic placeholders), and real navigation.

Key Takeaways
Plan your pages and sections before you build. An outline saves time
Keep file organization simple and name files descriptively
Build the skeleton first: structure, then content, then styling, then navigation
Use your tools (frameworks, AI assistants, templates) to accelerate, but understand what they produce

Week 03: From Local to Live
From Local to Live
Building and previewing your work locally is necessary, and deploying is what makes it real. Even when testing incomplete work privately, without sharing live deployment URLs (yet), deploying online is necessary. This page covers the deployment process to get your project from your computer to a live URL that anyone can visit.

Deployment isn't a magical final step you do one time when everything is perfect. It's something you do early and often. Over and over. A deployed project,  even a rough one, is fundamentally different from a local one because it exists in the world. You can share the project, test it on different devices, view it across different browsers and resolutions, and get feedback.

What Deployment Means
When you work on your project locally, that means it lives on your computer. Local is your computer. Remote is someone else's computer (e.g., a deployment/hosting platform like GitHub Pages or Netlify). You can see it in your browser, but nobody else can. Deployment is the process of putting your project on a server so it's accessible via a URL.

For this course, deployment means your project is:

Accessible via a public URL: Anyone with the link can see it
Hosted on a platform: Your files are served by a hosting provider, not your computer (your files are on someone else's computer, AKA deployed remotely)
Connected to your repository: Changes you push to GitHub can trigger updates to the deployed site/app
Deployment Platforms
You surveyed hosting options in Week 01. Here's a practical look at the platforms you're most likely to use:

GitHub Pages is the simplest option if your project is a static site (HTML, CSS, and JavaScript files with no server-side processing). You used GitHub Pages in earlier CMPA coursework. Enable it in your repository settings, point it to the right branch or folder, and your site is live. You can manually or automatically trigger GitHub Actions workflows to deploy your site.

Netlify deploys from a Git repository and offers more features than GitHub Pages, including custom build commands, form handling, serverless functions, and more. The free tier is very generous. Connect your GitHub repo, configure build settings, and deploy. Netlify provides excellent documentation and plenty of helpful starter templates/examples. 

Vercel is very similar to Netlify, with strong support for framework-based projects (primarily Next.js, though it also supports many other frameworks). Connect your GitHub repo, and it automatically detects your framework. You also have full control over a variety of deployment options, some of which are included in the free tier.

Cloudflare (Cloudflare Pages and Cloudflare Workers) offers extremely fast hosting with a simple setup. Like with other deployment platforms, you connect your GitHub repo, configure the build and deployment settings, and deploy. Cloudflare's free tier (for most of their offerings) is extraordinarily generous. Depending on your project, Pages might be ideal if your work focuses on static content, but Workers provides more capabilities.

All of these platforms provide automatic HTTPS, so your deployed site will be secure by default.

For most projects in this course, any of these platforms will work. Pick one and commit to it. You can always migrate later if needed/desired.

The Deployment Pipeline
A deployment pipeline is the series of steps from editing a file to seeing the change live on the web. Here's what that looks like:

Edit: You change a file in your code editor
Preview: You check the change locally in your browser
Commit: You save the change to version control (Git)
Push: You send the commit to GitHub
Deploy: The hosting platform detects the push and updates your live site (Note: this happens because you connected your GitHub repo to the hosting platform!)
With most platforms, steps 4 and 5 happen together. Push to GitHub, and within seconds to minutes, your live site reflects the change. This is called continuous deployment; every push automatically deploys.

Getting this pipeline working is the core goal of Exercise 03. Once it works, you have a reliable process for the rest of the semester: make changes, push, and they're live.

Deploying for the First Time
Your first deployment doesn't need to be impressive. It needs to work. Here's a practical sequence:

Make sure your project has an entry point. For most setups, this is an index.html file in the root of your project (or wherever your platform expects it: remember to read the docs for the platform/framework!).
Push your current project to GitHub. Commit all your files and push to the main branch.
Connect your hosting platform. Follow the platform's setup process to connect your GitHub repository.
Configure build settings. If you're using plain HTML/CSS, there's usually nothing to configure. If you're using a framework, specify the build command and output directory. Most of the platforms listed above will auto-detect popular frameworks and tools, so you might not need to configure much.
Deploy and test. Trigger a deployment (or let it happen automatically), then visit your deployment URL. Click through every page. Check it on your phone or tablet, too. Verify that it exists online, appears on your devices when you visit the URL, and that your deployment succeeded.
If something breaks, check the deployment logs on your hosting platform. The logs will almost always give you the clues you need to fix your deployment woes. Some common deployment issues:

Files not in the expected location (the platform is looking for index.html in the root, but you accidentally placed it in a subfolder)
Files/directories not named correctly or missing file extensions (e.g., you accidentally renamed index.html to My Project.jsx)
Build command is wrong (for framework projects)
Links use absolute paths that don't match the deployed URL (the absolute paths worked on your local computer, but not when deployed)
Version Control as a Habit
Deployment is connected to version control. Each deploy comes from a commit. This is a good thing! At first, it feels tedious. However, deploying after every commit means that every version of your deployed site is tracked, and you can always revert to a previous version if something breaks.

Good version control habits for this course:

Commit frequently. Don't wait until you think everything is "done" and perfect. Make progress, get something working, and make a small commit. Commit when you finish a section. Commit when you fix a bug. Commit when you reach a natural stopping point. Small, frequent commits are easier to understand and easier to undo.  

Write meaningful commit messages. "Updated stuff" says nothing. "Add header navigation to main template" tells you exactly what the commit is all about. Future You will appreciate the clarity and smaller commits made by Current You.

Push before you close your laptop (or turn off your PC). If your changes are only on your local machine, they're one storage drive failure away from being lost. Not likely? Maybe! It is good practice to push your local-only commits regularly. So, push to GitHub regularly.

Key Takeaways
Deploy early, and often, not when everything is "perfect" -> a live project is a real project
The deployment pipeline is: edit, preview, commit, push, deploy (rinse and repeat)
Pick a hosting platform and get continuous deployment working this week
Version control is the backbone of reliable deployment -> commit early, commit often, and push regularly
