Week 07: Refining Your Project
Refining Your Project
Refinement differs from building. While building is about adding (new pages, new features, new content, new underlying code, new deployment configuration, and on and on and on), refining is different. Refining is about improving what's already there.

 

This week, shift your focus from "what can I add?" to "how can I make what I have better?"

What Refinement Looks Like
 

Consistency
Look at your project as a whole:

Visual consistency. Do all pages use the same fonts, colors, and spacing? Are headings the same size and style across pages? Does navigation look and behave the same everywhere?
Content consistency. Is the written content consistent? What about images/videos/icons? Are similar elements described in similar ways? Do all pages feel like they belong to the same project?
Interaction consistency. Do interactive elements behave the same way throughout? If hovering over a card component shows a drop shadow on one page, does it do the same on every page?
Inconsistency makes a project feel unfinished even when all the pieces are present.

 

Completeness
Check for gaps:

Are there any pages or sections with placeholder content still in place?
Are all images actually loading? (Missing images show as broken icons or browser placeholder images. This is a common oversight, especially if the path to an image is slightly incorrect.)
Does every link lead somewhere? Do all navigation items work? (You might know where a broken link should lead, but a user most likely will not.)
Is every feature you claim your project has actually functional?
 

Usability
Test the experience:

Can a first-time visitor understand what your project is within a few seconds of accessing it?
Is the navigation clear? Can someone find any page within one or two clicks?
Does the project work on a phone? On a tablet? On a laptop or desktop computer? Not just "it loads," but is the project actually usable?
Are interactive elements discoverable? If something can be clicked/tapped, does it look like it can be clicked/tapped?
Is the text readable? Sufficient contrast? Appropriate font size(s)? Reasonable line length?
 

Performance
Check that your project loads well:

Are images an appropriate file size? Large, unoptimized images will slow everything down.
Does the site load without long delays?
Are there any errors? (Open your browser's developer tools and check the Console tab.)
 

Giving Good Peer Feedback
 

This week's discussion asks you to review a classmate's P01 deployment. Good feedback is specific, honest, and constructive:

Be specific. "The hero section on the home page explains your project's purpose clearly, but the grid of images took a long time to load on my phone" is specific and provides actionable feedback.

Be honest. If something doesn't work on your end, say that. Your classmate will benefit from knowing about an unknown-to-them problem.

Be constructive. Point out what works well. Do not spend all your time and attention pointing out problems. And, when you do identify an issue, try suggesting a potential direction they could explore if you can: "The navigation might work better on mobile and tablet if it didn't animate for 10-seconds every time a user taps on it" is far more helpful than "mobile nav broken. it's really bad."

Visit the site. Don't review from screenshots or descriptions. Go to the URL(s). Click/Tap around. Try it on different devices (phone, tablet, laptop, desktop).

Key Takeaways
Refinement is about improving what you have
Check for consistency, completeness, usability, and performance
Good peer feedback is specific, honest, and constructive


Week 07: Preparing for Final Delivery
Preparing for Final Delivery
P02 is due next week. This page is about getting your project ready for the final submission.

The P02 Delivery Package
 

Project 02 asks for three (3) things:

A deployed, functional project at a live URL
An expanded README.md in your GitHub repository
A REFLECTION.md in your GitHub repository
Most likely, the deployed site is where you've been focusing all of your efforts. However, your documentation is how you communicate what you built and what you learned. You'll need to focus some of your efforts on expanding the documentation.

Updating Your README
 

Your P01 README had four (4) sections: what it is, why it exists, what tools you used, and where to find it. For P02, you'll expand it:

Update existing sections: If your project has changed since P01 (which it did or should have), your description should reflect the current state. Updated tools? New features? Changed purpose? The README for your project needs to match the deployed project.

Add what changed from P01 to P02: Describe the evolution of your project. What did you add, fix, and improve? What did you cut and why? This section tells the story of your project's development.

This section is about decisions and rationale.

Writing Your Reflection
 

The REFLECTION.md is separate from the README. It's a personal, honest account of building your project. It should cover:

Decisions you made: What were the most significant decisions in your project? Tool choices? Feature priorities? Scope changes? How did you decide? How did those decisions play out?

What worked: Which aspects of your project, tools, or process worked well? What are you most satisfied with? Be specific.

What you would do differently: If you started this project over with what you know now, what would you change? Different tools? Different scope? Different approach? No focusing on regrets. Focus on how you would build this project differently if given the opportunity.

What you learned: What did you learn about building for the web? About using the tools you selected? About making decisions? About shipping? This can be technical, personal, or both.

Keep in mind that your reflection document is not a formal essay. So, do not write a formal essay. Please do not write a formal essay.

Write your reflection in a natural-to-you, honest voice. The length of your reflection should be whatever it takes for you to genuinely address these topics. That would most likely, typically, be somewhere in the range of 400-800 words, but the quality of your written reflection on building your project matters more than word count. 

Final Deployment Checklist
 

Before you submit P02, go through this list:

All content is real and complete. No placeholder text. No "coming soon" sections. No broken images. No broken videos. No navigation menus that take 10-seconds to animate.
All features work. Test every interactive element. Click every link. Submit every form. Test the calculator. Play the game again. Click through the image gallery. Verify that the features are working as they should.
The site works on mobile. Check on a real phone or use browser developer tools to simulate mobile.
The README is current. It reflects the deployed project, includes what changed from P01, and has a working link to the live site.
The REFLECTION.md is written. It addresses decisions, what worked, what you'd do differently, and what you learned.
The GitHub repository is clean. Files are organized. No unnecessary or leftover files.
The deployed URL works. Visit it in an incognito window. Make sure it loads correctly.
Key Takeaways
P02 delivery is three (3) parts: deployed site, expanded/updated README, and REFLECTION.md
Update your README to reflect the current project; document what changed from P01
Write an honest reflection about decisions, successes, and lessons learned; the reflection is not a formal essay
Use the final checklist before submitting
