# Project 02 Reflection

This project started as a practical tool for Brownsville Kuk Sool Won students, but it also became a lesson in how many decisions sit underneath a simple-looking website. The goal was to give students and families a clearer way to review rank requirements, track practice progress, and find instructor or study support without needing a complicated account system. By Project 02, the site included a landing page, student dashboard, contact actions, Weekly Focus access, a completed study-resource page, an expanded README, and this reflection.

One of the biggest decisions I made was to keep the project static. I used HTML, CSS, vanilla JavaScript, JSON, CSV, Node.js utility scripts, Git, GitHub, and GitHub Pages instead of a framework or backend. That choice kept deployment simple and made the files easier to inspect for class review. The tradeoff is that progress is stored locally in the browser with `localStorage`, so it is tied to the device and browser being used. That is not a perfect long-term system, but it fits the current scope because this version is a preparation and reference tool, not an official grading platform.

The part that worked best was turning curriculum data into something students can actually use. Instead of one long list of requirements, the dashboard lets a student choose a WKSA testing band and rank, then opens the material through grouped sections. The progress summary also helped the project feel more like a tool and less like a static reference page. Students can mark requirements as Untrained, In Progress, Trained, or Proficient, and the summary updates from that same data. I am satisfied with that because it connects the project to a real audience: students, parents, and instructors who need rank requirements to be easier to understand.

For P02, I spent more time looking at consistency and polish. I noticed that small issues made the project feel less finished: button text running outside boxes, mobile sections missing content that existed on desktop, copy like "Contact US" appearing in several places, and the dashboard requiring horizontal movement on mobile. Fixing those things was not as exciting as adding a new feature, but it made the experience better. The Weekly Focus and Contact Instructor areas were kept available in the mobile site flow, while redundant sidebar utilities were removed from secondary mobile pages so those pages could focus on their main task. Button sizing and focus states were tightened, contact card hover behavior was made more consistent, and the dashboard table was adjusted so the mobile view is easier to use.

I also made some scope decisions. The Study resources dropdown remains because organizing study resources was part of the original idea, but I limited it to completed curated resources and hid the per-row Study column for P02. I did not want the dashboard to look like it was full of incomplete links. I also removed the backend editor idea from the final project promise and hid the visible Save/Load progress controls. The import/export idea is still useful for a future version, but the workflow needs more polish before it should be shown as a student-facing feature. Those decisions helped the delivered version feel cleaner and more focused.

The most important technical correction from P01 to P02 was treating age band as part of the requirements logic. P01 proved that rank alone was not enough because junior and adult/youth students can have different requirement paths. Updating the data model and dashboard filtering around WKSA testing bands made the tool better match the way students actually prepare for testing.

If I started over, I would separate the dashboard JavaScript into smaller files much earlier. The single-file approach helped at the beginning because everything was in one place, but it became harder to reason about as the project grew. I would also define the final feature contract earlier so desktop and mobile could be checked against the same list from the start. On the CSS side, I would define responsibilities earlier: shared layout in `style.css`, active theme styling in `dragon.css`, and landing-page-specific layout in `home.css`. I eventually cleaned up unused legacy selectors and removed an unused alternate theme file, but I would build with that structure from the beginning next time.

I learned that web projects are not only about making things function. A feature can technically work and still feel unfinished if the layout shifts awkwardly, a button overflows, a page title does not match, or mobile users get a weaker version of the site. I also learned that documentation matters because it explains the decisions behind the work. The README is not just a class requirement; it is how someone else understands what the project is, what changed from P01 to P02, what was intentionally deferred, and what still needs attention.

The biggest lesson is that shipping means making choices. I could not build every feature fully, so I had to decide what to polish, what to defer, and what to explain honestly. A finished milestone is not perfect, but it should be functional, understandable, and honest about where it is going next. That is what I tried to do with this version of the Brownsville Kuk Sool Won Student Dashboard.
