# Project 02 Reflection

This project started as a practical tool for Brownsville Kuk Sool Won students, but it also became a lesson in how many decisions sit underneath a simple-looking website. The main decision I made was to keep the project static: HTML, CSS, and JavaScript without a framework or backend. That choice kept deployment simple through GitHub Pages and made the files easier to inspect for class review. The tradeoff is that all progress is browser-local, and the dashboard logic has to do a lot of work on the client side.

The part that worked best was turning curriculum data into something students can actually use. Instead of one long list of requirements, the dashboard lets a student choose a testing band and rank, then opens the material through grouped sections. The progress summary also helped the project feel more like a tool and less like a static reference page. I am also satisfied with the way the project stayed connected to a real audience. The decisions were not abstract; they were about what would make sense to students, parents, and instructors.

For P02, I spent more time looking at consistency and polish. I noticed that small issues made the project feel less finished: button text running outside boxes, mobile sections missing content that existed on desktop, copy like “Contact US” appearing in several places, and the dashboard requiring horizontal movement on mobile. Fixing those things was not as exciting as adding a new feature, but it made the experience better. I also decided to hide the visible save/load progress controls for now. The code is still there, but the interface needed to feel cleaner for this delivery. I would rather bring that feature back later when I can present it clearly than leave confusing controls on screen.

If I started over, I would separate the dashboard JavaScript into smaller files much earlier. The single-file approach helped at the beginning because everything was in one place, but it became harder to reason about as the project grew. I would also plan the study-resource pages earlier. The Coming Soon page is now an intentional destination, not an accidental placeholder, but a stronger content schedule would make those links more valuable.

I learned that web projects are not only about making things function. A feature can technically work and still feel unfinished if the layout shifts awkwardly, a button overflows, a page title does not match, or mobile users get a weaker version of the site. I also learned that documentation matters because it explains the decisions behind the work. The README is not just a class requirement; it is how someone else understands what the project is, what changed, and what still needs attention.

The biggest lesson is that shipping means making choices. I could not build every feature fully, so I had to decide what to polish, what to defer, and what to explain honestly. That is probably the most useful thing I learned from this project: a finished milestone is not perfect, but it should be functional, understandable, and honest about where it is going next.
